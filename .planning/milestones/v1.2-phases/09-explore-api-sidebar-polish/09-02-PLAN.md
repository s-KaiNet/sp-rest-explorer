  **Step 1: Ensure root children carry returnType (use-api-navigation.ts)**

  The root children in `useApiNavigation()` already include `returnType` in the ChildEntry (line 63: `returnType: fn.returnType`). This is the entity fullName (e.g., "SP.Web", "SP.Publishing.Pages.SitePage") which we'll use to extract namespaces. No changes needed here — verify this is present.

  **Step 2: Implement namespace grouping in Sidebar.tsx**

  Transform Sidebar.tsx to support two modes: `variant='root'` (namespace-grouped) and `variant='default'` (flat list, unchanged).

  When `variant='root'`:
  1. Group entries by namespace extracted from `entry.returnType`:
     - Parse namespace: take everything up to the last `.` in returnType (e.g., "SP.Web" → "SP", "SP.Publishing.Pages.SitePage" → "SP.Publishing.Pages")
     - Entries with no returnType or no `.` in returnType → "Core" group
  2. Sort groups: "Core" always first, then alphabetically by namespace name
  3. Sort items within each group alphabetically by name
  4. Compute stripped display names: remove common namespace prefix from item names where the item name starts with the namespace (e.g., in "SP.Social" group, an item named "SP.Social.SocialPost" shows as "SocialPost"). NOTE: Many root function names do NOT include the namespace prefix (they're just function names like "GetByTitle"), so stripping only applies when the name actually starts with the namespace + ".".
  5. Render collapsible groups using the SAME visual pattern as TypesSidebar.tsx:
     - Import `{ useState } from 'react'` and `{ ChevronRight, ChevronDown } from 'lucide-react'`
     - Track collapsed groups via `useState<Set<string>>(new Set())` — empty set = all expanded (same pattern as TypesSidebar)
     - Group header: button with chevron + uppercase namespace name, same classes as TypesSidebar group header
     - Subtle separator between groups (not on first): `<div className="my-1 border-t border-border/30" />`
     - Items inside collapsed groups are hidden
  6. Reset collapsed state: The Sidebar component is remounted via key-based SidebarTransition when navigating, so collapsed state naturally resets (new Sidebar instance = fresh useState). This satisfies the "resets to all-expanded when returning to root" requirement without extra logic.

  When `variant='default'`: Keep existing behavior — split into navProperties/functions with separator.

  **Important implementation notes:**
  - Do NOT import/reuse TypesSidebar directly — it's typed for EntityType[], not ChildEntry[]. Instead, replicate the visual pattern (same Tailwind classes, same chevron icons, same collapsible behavior) inside Sidebar.tsx.
  - The "Core" label is specific to Explore API per user decision. TypesSidebar uses "Other" / "No Group" for its ungrouped items — do not change that.
  - Group headers show NO item count — just namespace name + chevron.

  **Step 3: Reposition root indicator badge in SidebarItem.tsx**

  Currently (variant='root'), the `<>` badge is on the LEFT side (before the name span). Move it to the RIGHT side (after the name span), matching the position of FN/NAV badges in variant='default'.

  Change from:
  ```
  {variant === 'root' && (<badge />)}
  <span className="min-w-0 flex-1 truncate">{entry.name}</span>

  {variant === 'root' && (<badge />)}
  {variant === 'default' && showTypeTags && (...badges...)}
  ```

  Also add a `displayName` prop to SidebarItem (optional, defaults to `entry.name`) so Sidebar.tsx can pass the stripped name for namespace-grouped items.

  Update the SidebarItem interface:

    onClick: () => void
    showTypeTags?: boolean
    variant?: 'default' | 'root'
    displayName?: string  // Stripped name for namespace groups
  }

  Render `displayName ?? entry.name` in the truncate span. Keep `entry.name` in the `title` attribute (full name on hover).

<task type="auto">
  <name>Task 2: Fix horizontal scrollbar flickering during slide animation</name>
  <files>app/src/pages/ExplorePage.tsx, app/src/components/navigation/NavigationTransition.tsx</files>
  <action>
  The horizontal scrollbar flickers during forward navigation because the `slide-in-from-right` animation temporarily positions content off-screen to the right, causing the scroll container to expand horizontally.

  **Fix 1: Add overflow-x-hidden to the sidebar scroll container in ExplorePage.tsx**

  The sidebar content scroll container (line 102) currently has `className="flex-1 overflow-y-auto"`. Add `overflow-x-hidden` to prevent horizontal scrollbar:
  ```
  className="flex-1 overflow-y-auto overflow-x-hidden"
  ```

  Note: ResizablePanel.tsx already has `overflow-x-hidden` on its inner div (line 67), but the scroll container INSIDE it (in ExplorePage.tsx) also needs it because `overflow-y-auto` creates a new scroll context.

  **Fix 2: Ensure NavigationTransition doesn't cause overflow**

  In NavigationTransition.tsx, the SidebarTransition wrapper div has `style={{ minHeight: '100%' }}`. This is fine. The animation classes (`animate-in slide-in-from-right`) are from tw-animate-css and use `translateX`. The overflow-x-hidden on the parent scroll container will clip the animated element during transition, preventing the scrollbar.

  No changes needed to NavigationTransition.tsx itself — the fix in ExplorePage.tsx is sufficient.

  **Fix 3: Reset scroll position on navigation**

  Per user decision: "Scroll position resets to top when navigating to deeper levels."

  In ExplorePage.tsx, add a ref to the sidebar scroll container and reset scrollTop when pathKey changes:

  ```typescript
  const sidebarScrollRef = useRef<HTMLDivElement>(null)

  // Reset sidebar scroll position on navigation
  useEffect(() => {
    if (sidebarScrollRef.current) {
      sidebarScrollRef.current.scrollTop = 0
    }
  }, [pathKey])
  ```

  Apply the ref to the scroll container div:
  ```
  <div ref={sidebarScrollRef} className="flex-1 overflow-y-auto overflow-x-hidden">
  ```

  Per user decision: Keep existing animation speed and easing — only fix the scrollbar issue. Do NOT change animation timing or add new animations.
  </action>
