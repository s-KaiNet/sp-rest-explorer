  <name>Task 1: Create shared handler with structured logging and stage timing</name>
  <files>backend/src/handlers/metadata-handler.ts</files>
  <action>
Create `backend/src/handlers/metadata-handler.ts` with:

1. **`HandlerResult` interface** (EXPORTED):
   ```typescript
   export interface HandlerResult {
     status: 'ok' | 'error';
     duration: number;
     blobsUploaded: number;
     stages: Record<string, { duration: number }>;
     error?: string;
     failedStage?: string;
   }
   ```

2. **`handleMetadataGeneration(context: InvocationContext): Promise<HandlerResult>`** (EXPORTED):

   The orchestration flow:

   a. Record `overallStart = Date.now()`
   b. Initialize `stageTimings: Record<string, { duration: number }> = {}`

   c. **Auth stage:**
      - `start = Date.now()`
      - `const token = await getToken()`
      - Read `SP_URL` from `process.env.SP_URL` — throw if missing (follow existing pattern in `auth.ts`)
      - `stageTimings.auth = { duration: Date.now() - start }`
      - `context.log('[stage:auth] completed in ' + stageTimings.auth.duration + 'ms')`

   d. **Pipeline stage (fetch + parse + compress):**
      - `start = Date.now()`
      - `const result = await runPipeline(token, spUrl)` — if this throws (fetch failed after Phase 19's retries), it propagates out of the try block → no blobs written (OPS-07 all-or-nothing)
      - `stageTimings.pipeline = { duration: Date.now() - start }`
      - `context.log('[stage:pipeline] completed in ' + stageTimings.pipeline.duration + 'ms')`

   e. **Upload stage:**
      - `const now = new Date()` — fresh date per invocation, INSIDE handler, NOT module scope (OPS-03)
      - `start = Date.now()`
      - `await uploadBlobs(context, result, now)`
      - `stageTimings.upload = { duration: Date.now() - start }`
      - `context.log('[stage:upload] completed in ' + stageTimings.upload.duration + 'ms')`

   f. **Summary:**
      - `const duration = Date.now() - overallStart`
      - `context.log('Pipeline complete: 6 blobs uploaded in ' + duration + 'ms')`
      - Return `{ status: 'ok', duration, blobsUploaded: 6, stages: stageTimings }`

   g. **Error handling (catch block):**
      - Determine which stage failed based on which stageTimings are present (if auth missing → 'auth', if pipeline missing → 'pipeline', otherwise → 'upload')
      - `context.error('[error] Pipeline failed at stage: ' + failedStage + ' — ' + message)`
      - Log durations of all stages that DID complete: iterate stageTimings and log each
      - Re-throw the error so Azure Functions retry policy can activate for timer trigger (OPS-06)

   h. **Also construct the error HandlerResult** before re-throwing — attach it to the error object or use a pattern where the HTTP trigger can catch and extract it. Recommended approach: throw a custom error that carries `HandlerResult` data, OR use a wrapper in the HTTP trigger that catches and builds the error response.

      Simpler approach (recommended): The handler always throws on error. The HTTP trigger wraps the call in try/catch and builds the error response there. The timer trigger lets the throw propagate for retry. The handler's catch block handles logging only, then re-throws.

   i. Imports:
      - `{ InvocationContext } from '@azure/functions'`
      - `{ getToken } from '../auth.js'`
      - `{ runPipeline } from '../pipeline/index.js'`
      - `{ uploadBlobs } from '../upload/blob-uploader.js'`

   j. If `context.retryContext` exists, log retry information: `[retry] attempt ${context.retryContext.retryCount} of ${context.retryContext.maxRetryCount}`
  </action>

  <name>Task 2: Create timer trigger, HTTP trigger, and update local settings</name>
  <files>backend/src/functions/generateMetadata.ts, backend/src/functions/generateMetadataHttp.ts, backend/local.settings.json</files>
  <action>
1. **Create `backend/src/functions/generateMetadata.ts`** (timer trigger):

   Follow the existing `validateAuth.ts` pattern for structure:

   ```typescript
   import 'dotenv/config';
   import { app, InvocationContext, Timer } from '@azure/functions';
   import { handleMetadataGeneration } from '../handlers/metadata-handler.js';
   ```

   a. Handler function `generateMetadata(timer: Timer, context: InvocationContext): Promise<void>`:
      - `context.log('Timer-triggered metadata generation started')`
      - If `timer.isPastDue`: `context.log('Timer is past due — running immediately')`
      - `await handleMetadataGeneration(context)` — let errors propagate for retry

   b. Register with `app.timer('generateMetadata', { ... })`:
      - `schedule: '%TIMER_SCHEDULE%'` — reads from app setting (OPS-02). The `%` syntax is Azure Functions' way to bind to app settings.
      - `handler: generateMetadata`
      - `retry` policy (OPS-06):
        ```typescript
        retry: {
          strategy: 'fixedDelay',
          maxRetryCount: 2,
          delayInterval: { minutes: 5 },
        }
        ```

2. **Create `backend/src/functions/generateMetadataHttp.ts`** (HTTP trigger):

   ```typescript
   import 'dotenv/config';
   import { app, HttpRequest, HttpResponseInit, InvocationContext } from '@azure/functions';
   import { handleMetadataGeneration } from '../handlers/metadata-handler.js';
   ```

   a. Handler function `generateMetadataHttp(request: HttpRequest, context: InvocationContext): Promise<HttpResponseInit>`:
      - `context.log('HTTP-triggered metadata generation started')`
      - Try: `const result = await handleMetadataGeneration(context)`
        - Return `{ status: 200, jsonBody: result }`
      - Catch: Build error response per user decision:
        - Extract error message
        - Return `{ status: 500, jsonBody: { status: 'error', stage: failedStage, error: message, stageTimings: { ... } } }`
        - For the stage timings in error response: the handler logs them via context.log, but to include in HTTP response, the catch needs the timing data. Use this pattern: have the handler's catch block attach stageTimings to the error object before re-throwing, e.g., `(error as any).stageTimings = stageTimings; (error as any).failedStage = failedStage;`. Then the HTTP trigger's catch can extract them.

   b. Register with `app.http('generateMetadataHttp', { ... })`:
      - `methods: ['POST']`
      - `authLevel: 'function'` (OPS-05)
      - `handler: generateMetadataHttp`
      - Do NOT add retry policy to HTTP trigger (research pitfall: HTTP triggers don't support function-level retry)

3. **Update `backend/local.settings.json`:**

   Read the existing file first. Add `TIMER_SCHEDULE` to the `Values` section:
   - `"TIMER_SCHEDULE": "0 0 1 * * *"` (default: 1 AM UTC daily, per OPS-01)
   - Also ensure `AzureWebJobsStorage` is present (set to `"UseDevelopmentStorage=true"` if not already there — needed for blob SDK in local dev)
   - Keep all existing values (SP_URL, ENTRA_CLIENT_ID, etc.)
  </action>
