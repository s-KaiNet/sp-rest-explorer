**Step 1 — Build locally first (Claude does this):**
```bash
cd backend && npm run build
```
Verify: dist/ directory has compiled JS files, zero TypeScript errors.

**Step 2 — Deploy to Azure:**
```bash
cd backend && npm run deploy
```
This runs: clean → build → `func azure functionapp publish sp-rest-explorer-new`

If you get auth errors, run `az login` first. If the Function App doesn't exist, it was supposed to be pre-created as `sp-rest-explorer-new`.

**Step 3 — Configure app settings via CLI:**

Run these commands (fill in actual values from your Azure environment):

```bash
az functionapp config appsettings set \
  --name sp-rest-explorer-new \
  --resource-group <YOUR_RESOURCE_GROUP> \
  --settings \
    ENTRA_TENANT_ID="<your-tenant-id>" \
    ENTRA_CLIENT_ID="<your-client-id>" \
    SP_URL="<your-sharepoint-url>" \
    TIMER_SCHEDULE="0 0 1 * * *"
```

For certificate PEM values (multiline — use Azure Portal or separate commands):
```bash
az functionapp config appsettings set \
  --name sp-rest-explorer-new \
  --resource-group <YOUR_RESOURCE_GROUP> \
  --settings SP_CERT_PEM="<pem-certificate-content>"

az functionapp config appsettings set \
  --name sp-rest-explorer-new \
  --resource-group <YOUR_RESOURCE_GROUP> \
  --settings SP_KEY_PEM="<pem-private-key-content>"
```

For the storage connection string (from `sprestapiexplorernew` storage account):
```bash
az functionapp config appsettings set \
  --name sp-rest-explorer-new \
  --resource-group <YOUR_RESOURCE_GROUP> \
  --settings AzureWebJobsStorage="<connection-string-for-sprestapiexplorernew>"
```

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 3: Trigger function and validate production blobs</name>
  <what-built>Function is deployed and configured. Now validate it works in production.</what-built>
  <how-to-verify>

**Step 1 — Trigger via HTTP endpoint:**

Get the function URL from Azure Portal → Function App → Functions → generateMetadataHttp → Get Function Url (with function key).

```bash
curl -X POST "https://sp-rest-explorer-new.azurewebsites.net/api/generateMetadataHttp?code=<FUNCTION_KEY>"
```

Expected: JSON response with stage timings (auth, fetch, parse, compress, upload) and success status.

**Step 2 — Verify 6 blobs exist:**

Check `api-files` container on `sprestapiexplorernew` storage account. All 6 blobs should be present:

1. `metadata.latest.json` — full JSON metadata
2. `metadata.latest.xml` — raw XML from SharePoint  
3. `metadata.latest.zip.json` — lz-string compressed JSON
4. `{year}y_m{month}_metadata.json` — monthly snapshot (e.g., `2026y_m2_metadata.json`)
5. `{year}y_m{month}_metadata.xml` — monthly XML snapshot
6. `{year}y_m{month}_metadata.zip.json` — monthly compressed snapshot

```bash
az storage blob list \
  --account-name sprestapiexplorernew \
  --container-name api-files \
  --output table
```

**Step 3 — Validate blob content format:**

Download `metadata.latest.json` and spot-check the structure:
- Has `entities` array with EntityType objects (Name, Properties, NavigationProperties)
- Has `functions` array with FunctionImport objects
- Has `associations` array
- JSON structure matches what the frontend currently consumes from production

```bash
az storage blob download \
  --account-name sprestapiexplorernew \
  --container-name api-files \
  --name metadata.latest.json \
  --file /tmp/metadata-check.json

# Check structure
node -e "const d=require('/tmp/metadata-check.json'); console.log('entities:', d.entities?.length, 'functions:', d.functions?.length, 'associations:', d.associations?.length)"
```
  </how-to-verify>
