# Phase 07.1: Fix Search Experience - Research

**Researched:** 2026-02-14
**Domain:** Client-side search indexing of hierarchical API trees with grouped result display
**Confidence:** HIGH

## Summary

This phase restructures the search system from three groups (entities, functions, nav properties) to two groups (Entities, API Endpoints) with reliable navigation. The primary technical challenge — walking the API tree from `_api/` root functions to index all reachable endpoints — turns out to be trivially fast. Empirical analysis of the actual metadata (2,449 entities, 3,528 functions) shows the BFS tree walk completes in **~1.2ms** with **3,330 entries** and naturally converges at depth 8 due to cycle detection. No depth limit, backend pre-computation, or Web Worker is needed.

The existing MiniSearch 7.2.0 + cmdk 1.1.1 stack handles the projected ~5,779 documents (2,449 entities + 3,330 endpoints) with sub-2ms search times. The main implementation work is: (1) rewriting the search index builder to use the BFS tree-walk for API endpoints, (2) restructuring the `CommandPalette` component from 3 groups to 2 with collapsible "Show more" expansion, and (3) fixing entity navigation to use `/entity/<fullName>` instead of the broken `entityPathMap` approach.

**Primary recommendation:** Run BFS tree-walk to convergence (no depth limit) on metadata boot, store results in a single MiniSearch index with a `kind` field, group results client-side using MiniSearch's `filter` option or post-search JS filtering.

### Pattern 1: BFS Tree-Walk to Convergence
**What:** Walk the API tree starting from root functions, following composable functions and nav properties through entity types, using a visited-entities Set for cycle detection.
**When to use:** Building the API Endpoints search index on app load.
**Key insight:** The tree naturally converges at depth 8 with entity-level cycle detection. No depth limit is needed — the visited-entities Set prevents infinite loops and guarantees termination.

```typescript
// Source: Empirical analysis of actual metadata (2026-02-14)
interface EndpointEntry {
  id: string           // "endpoint:{parentEntity}|{leafName}|{kind}" for dedup
  name: string         // Leaf name (searchable)
  path: string         // Full _api/... path (displayed, not searched)
  kind: 'function' | 'navProperty'
  parentEntity: string // Entity full name the leaf belongs to
  isRoot: boolean      // Whether this is a root function
}

function buildApiEndpointIndex(
  metadata: Metadata,
  lookupMaps: LookupMaps,
): EndpointEntry[] {
  const { entityByFullName, functionById } = lookupMaps
  const indexed = new Map<string, EndpointEntry>()  // identity -> entry (dedup)
  const visitedEntities = new Set<string>()          // cycle detection
  const queue: Array<{ entityFullName: string; path: string }> = []

  // Seed: ALL root functions (composable + non-composable)
  for (const fn of Object.values(metadata.functions)) {
    if (!fn.isRoot) continue
    const identity = `ROOT|${fn.name}|function`
    indexed.set(identity, {
      id: identity,
      name: fn.name,
      path: `_api/${fn.name}`,
      kind: 'function',
      parentEntity: '',
      isRoot: true,
    })
    // Only traverse children of composable root functions
    if (fn.isComposable && fn.returnType && entityByFullName.has(fn.returnType)) {
      if (!visitedEntities.has(fn.returnType)) {
        visitedEntities.add(fn.returnType)
        queue.push({ entityFullName: fn.returnType, path: `_api/${fn.name}` })
      }
    }
  }

  // BFS until convergence
  while (queue.length > 0) {
    const { entityFullName, path } = queue.shift()!
    const entity = entityByFullName.get(entityFullName)
    if (!entity) continue

    // Nav properties
    for (const nav of entity.navigationProperties) {
      const identity = `${entityFullName}|${nav.name}|navProperty`
      if (!indexed.has(identity)) {
        indexed.set(identity, {
          id: identity,
          name: nav.name,
          path: `${path}/${nav.name}`,
          kind: 'navProperty',
          parentEntity: entityFullName,
          isRoot: false,
        })
      }
      // Nav properties always point to an entity — continue if not visited
      if (entityByFullName.has(nav.typeName) && !visitedEntities.has(nav.typeName)) {
        visitedEntities.add(nav.typeName)
        queue.push({ entityFullName: nav.typeName, path: `${path}/${nav.name}` })
      }
    }

    // Bound functions
    for (const fnId of entity.functionIds) {
      const fn = functionById.get(fnId)
      if (!fn) continue
      const identity = `${entityFullName}|${fn.name}|function`
      if (!indexed.has(identity)) {
        indexed.set(identity, {
          id: identity,
          name: fn.name,
          path: `${path}/${fn.name}`,
          kind: 'function',
          parentEntity: entityFullName,
          isRoot: false,
        })
      }
      // Only traverse composable functions
      if (fn.isComposable && fn.returnType && entityByFullName.has(fn.returnType) && !visitedEntities.has(fn.returnType)) {
        visitedEntities.add(fn.returnType)
        queue.push({ entityFullName: fn.returnType, path: `${path}/${fn.name}` })
      }
    }
  }

  return Array.from(indexed.values())
}
```

### Pattern 2: Single MiniSearch Index with Kind-Based Grouping
**What:** One MiniSearch index containing both entity documents and endpoint documents, differentiated by a `kind` stored field. Group results at display time using JS `filter()`.
**When to use:** Always — simpler than maintaining two separate indexes.

```typescript
// Build the combined index
const index = new MiniSearch<SearchDocument>({
  fields: ['name'],           // Only search the leaf name
  storeFields: ['name', 'fullName', 'kind', 'path', 'parentEntity', 'isRoot', 'endpointKind'],
  searchOptions: {
    prefix: true,
    fuzzy: 0.2,
    boost: { name: 2 },
  },
  tokenize: (text) => text.split(/[\s._]+/),
})

// Add entity documents (ALL entities including Collection() types)
for (const entity of Object.values(metadata.entities)) {
  index.add({
    id: `entity:${entity.fullName}`,
    name: entity.name,
    fullName: entity.fullName,
    kind: 'entity',
  })
}

// Add endpoint documents from tree walk
for (const entry of endpointEntries) {
  index.add({
    id: `endpoint:${entry.id}`,
    name: entry.name,
    kind: 'endpoint',
    path: entry.path,
    parentEntity: entry.parentEntity,
    isRoot: entry.isRoot,
    endpointKind: entry.kind,  // 'function' | 'navProperty'
  })
}

// At search time — group results
const results = index.search(query)
const entities = results.filter(r => r.kind === 'entity')
const endpoints = results.filter(r => r.kind === 'endpoint')
```

### Pattern 3: Collapsible Group with "Show more" in cmdk
**What:** Render a limited number of items initially per group, with a "Show more (N remaining)" button that expands the group. Use simple React state — no external library needed.
**When to use:** For the command palette result groups.

```tsx
// Collapsible group with Show more
const INITIAL_SHOW = 5  // Show first 5 results per group

function SearchGroup({
  heading,
  results,
  renderItem,
}: {
  heading: string
  results: SearchResult[]
  renderItem: (result: SearchResult) => React.ReactNode
}) {
  const [expanded, setExpanded] = useState(false)
  const [collapsed, setCollapsed] = useState(false)

  if (results.length === 0) return null
  
  const visible = expanded ? results : results.slice(0, INITIAL_SHOW)
  const remaining = results.length - INITIAL_SHOW

  if (collapsed) {
    return (
      <div
        className="cursor-pointer px-2 py-1.5 text-xs text-muted-foreground hover:text-foreground"
        onClick={() => setCollapsed(false)}
      >
        {heading} ({results.length}) ▶
      </div>
    )
  }

  return (
    <CommandGroup heading={heading}>
      {visible.map(renderItem)}
      {remaining > 0 && !expanded && (
        <div
          className="cursor-pointer px-2 py-1.5 text-xs text-muted-foreground hover:text-foreground"
          onClick={() => setExpanded(true)}
        >
          Show {remaining} more...
        </div>
      )}
    </CommandGroup>
  )
}
```

### Anti-Patterns to Avoid
- **Don't search path text for endpoints:** CONTEXT.md explicitly says "path is displayed but NOT searched." Only the leaf name should be in MiniSearch `fields`. Store `path` in `storeFields` for display only.
- **Don't use entityPathMap for entity navigation:** The current approach tries to resolve entities to `_api/` paths, which only works for ~265 root-reachable entities out of 1,928 total. Entity detail pages exist at `/entity/<fullName>` and work for ALL entities.
- **Don't use depth limits in the tree walk:** The tree converges naturally at depth 8. Adding a configurable depth limit adds complexity without benefit.
- **Don't build two MiniSearch indexes:** One index with kind-based filtering is simpler and equally performant.
- **Don't use forceMount on cmdk groups:** When `shouldFilter={false}`, cmdk already shows whatever you render. Empty groups should be handled by conditional rendering, not forceMount.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cycle detection in tree walk | Custom depth tracking per path | `visitedEntities: Set<string>` — entity-level visited set | Entity-level dedup is sufficient: if an entity type has been visited, all its children are already indexed. Only 7 true duplicates exist across paths. Path-level tracking would add complexity for zero benefit. |
| Full-text search | Custom substring matching | MiniSearch with `prefix: true, fuzzy: 0.2` | MiniSearch handles tokenization, scoring, ranking, prefix matching, and fuzzy matching. Hand-rolling any of this is worse. |
| Keyboard navigation across groups | Custom key handlers | cmdk's built-in `loop` prop + `CommandGroup` + `CommandItem` | cmdk already handles arrow key navigation seamlessly across groups, skipping group headings automatically. |
| Collection type handling | Custom unwrapping logic | Use `entityByFullName` directly — Collection() types are real entities in the metadata | The metadata includes `Collection(SP.Alert)` as a standalone entity with its own functions. No unwrapping needed for the tree walk. |
| Debounce | Custom setTimeout management | Keep existing 120ms debounce pattern | Already implemented correctly with cleanup. |

**Key insight:** The existing metadata structure already handles the hard problems (Collection types as entities, function-to-entity binding via returnType, composability flags). The tree walk is just BFS with a visited set — the complexity is in the metadata model, which is already solved.

## Common Pitfalls

### Pitfall 1: Entity Navigation Route Mismatch
**What goes wrong:** CONTEXT.md says "Clicking navigates to `/entities/<name>`" but the actual route in `routes.tsx` is `/entity/:typeName` (singular, and the parameter is `fullName` not `name`).
**Why it happens:** Discussion used approximate route names.
**How to avoid:** Navigate to `/entity/${encodeURIComponent(entity.fullName)}` — this matches the existing `TypesPage` route and how `TypeLink` already works.
**Warning signs:** Entity search results navigate to 404 pages.

### Pitfall 2: Collection() Entity Types in Entity Search
**What goes wrong:** Collection() entities (521 of them) appear alongside regular entities in search results. They are real entity types with bound functions (e.g., `Collection(SP.Alert)` has `GetById`, `Add`, etc.).
**Why it happens:** `metadata.entities` includes both regular entities (SP.Web) and collection wrappers (Collection(SP.Web)).
**How to handle:** Include ALL entities in the search index, including Collection() types. They are legitimate entity types that users may want to find. The full OData name displayed subtly makes them distinguishable.
**Note:** Searching "Alert" will show both "SP.Alert" and "Collection(SP.Alert)" — both are valid, navigable entity types.

### Pitfall 3: Non-Composable Root Functions as Leaves
**What goes wrong:** Non-composable root functions (478 of them) don't have children to traverse, but they ARE valid `_api/` endpoints that should appear in search results.
**Why it happens:** The composability check prevents traversal of their children, but the functions themselves must still be indexed.
**How to avoid:** Index ALL root functions (composable + non-composable). Only the composability check gates child traversal, not the root function's own indexing.
**Warning signs:** Searching for a non-composable root like "AddMember" returns no results.

### Pitfall 4: Path Prefix Inconsistency
**What goes wrong:** Some code uses `/_api/` (with leading slash) and some uses `_api/` (without). The route in `routes.tsx` is `_api/*` (no leading slash) but `useApiNavigation` builds paths with `/_api/` (leading slash).
**Why it happens:** React Router hash routing handles both, but consistency matters for `navigate()`.
**How to avoid:** Use `/_api/` (with leading slash) consistently for all navigate() calls, matching the existing `useApiNavigation` convention.
**Warning signs:** Navigation works from sidebar but not from search results (or vice versa).

### Pitfall 5: buildEntityPathMap is No Longer Needed
**What goes wrong:** The existing `buildEntityPathMap` function in `CommandPalette.tsx` becomes dead code but isn't removed, causing confusion.
**Why it happens:** The new architecture eliminates the need for this function — entities navigate to `/entity/<fullName>` and endpoints store their own resolvable `_api/` paths.
**How to avoid:** Remove `buildEntityPathMap` and `resolveSearchResultPath` entirely. Replace with direct path lookups from the search result's stored fields.
**Warning signs:** Old path resolution code is still referenced but never executes correctly.

### Pitfall 6: Search Index Built Before Lookup Maps
**What goes wrong:** The new tree-walk depends on `LookupMaps` (entityByFullName, functionById, entityChildren), but if `initSearchIndex` is called before `initLookupMaps`, the tree walk fails.
**Why it happens:** Boot sequence timing.
**How to avoid:** The current `boot.ts` already calls `initLookupMaps(data)` before `initSearchIndex(data)`. Maintain this order. The new `buildSearchIndex` should accept the `LookupMaps` as a parameter.
**Warning signs:** Empty endpoint search results despite metadata being loaded.

### MiniSearch Configuration for Two-Group Search
```typescript
// Source: Context7 MiniSearch docs + verified against actual data

const index = new MiniSearch<SearchDocument>({
  // Only index the leaf name — NOT path, NOT fullName for endpoints
  fields: ['name'],
  // Store everything needed for display and navigation
  storeFields: ['name', 'fullName', 'kind', 'path', 'parentEntity', 'isRoot', 'endpointKind'],
  searchOptions: {
    prefix: true,     // "Web" matches "WebParts"
    fuzzy: 0.2,       // Typo tolerance
  },
  // Tokenize on word boundaries, dots, and underscores
  tokenize: (text) => text.split(/[\s._]+/),
})
```

**Note on entity fullName searchability:** For entities, both `name` (e.g., "Web") and `fullName` (e.g., "SP.Web") should be searchable. The tokenizer splits on dots, so indexing just `name` as the field but also adding fullName as a second indexed field for entities achieves this. Alternatively, keep `fullName` as a searchable field for entities but NOT for endpoints.

**Recommended approach:** Use two fields `['name', 'fullName']` in the index, but for endpoint docs set `fullName` to `''` (empty) so it doesn't contribute to search. Or use a custom `extractField` to return empty string for endpoints' fullName.
