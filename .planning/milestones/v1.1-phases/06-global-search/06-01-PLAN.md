  <action>
    Create `app/src/components/search/CommandPalette.tsx` — the full command palette component.

    **Component interface:**
    ```tsx
    interface CommandPaletteProps {
      open: boolean
      onOpenChange: (open: boolean) => void
      onSelect: (path: string) => void
    }
    ```

    **Implementation details (per user decisions in CONTEXT.md):**

    1. **Structure:** Use `CommandDialog` from shadcn (wraps cmdk + Radix Dialog). Add `@radix-ui/react-visually-hidden` `VisuallyHidden` around the `DialogTitle` for a11y (command dialogs need a title but we don't want to show one visually). Add `DialogDescription` also hidden via VisuallyHidden.

    2. **Search input:** The `CommandInput` from cmdk handles this. Set placeholder to "Type to search entities, functions, properties..."

    3. **Search execution:**
       - Import `getSearchIndex` from `@/lib/metadata`
       - Use `useState` for the query string
       - Use `useMemo` or `useEffect` + `useState` for search results
       - Only fire search when query length >= 2 (per decision: "Minimum 2 characters required before search fires")
       - When query is 1 char: show hint "Type 2+ characters to search"
       - When query is empty: show placeholder text only
       - Apply 100-150ms debounce on the search input using a simple `setTimeout`/`useRef` pattern (no external debounce library)
       - Call `searchIndex.search(query, { limit: 21 })` — this returns SearchResult objects with `id`, `score`, and stored fields (`name`, `fullName`, `kind`, `parentEntity`)

    4. **Result grouping:**
       - Group results by `kind`: 'entity', 'function', 'navProperty'
       - Cap at 7 results per group (max ~21 visible)
       - Use `CommandGroup` with `heading` prop for each group
       - Group headings: "Entities", "Functions", "Nav Properties"
       - Hide empty groups (only render groups that have results — this is natural since we just skip empty arrays)
       - Always show group headings even when only one group has matches (just render the groups that have items)

    5. **Result items (CommandItem):**
       - Each item shows a kind-specific icon on the left:
         - Entity: reuse the `<>` icon pattern from SidebarItem (green bg-type-entity/10 text-type-entity)
         - Function: `ƒ` icon (blue bg-type-fn/10 text-type-fn)
         - Nav Property: `NAV` icon (purple bg-type-nav/10 text-type-nav)
       - Main text: The result `name` with query text highlighted in bold. Implement highlight by splitting the name on the query string (case-insensitive) and wrapping matches in `<mark>` or `<strong>` with appropriate styling.
       - Below the name: parent breadcrumb trail for disambiguation (e.g., "SP.List › Functions"). Build from `parentEntity` field + kind. Format: `{parentEntityShortName} › {kindLabel}`. If no parentEntity (root function), show "Root" label.
       - Root functions: show an explicit "Root" badge (small pill: `text-[10px] rounded bg-type-entity/10 px-1.5 py-0.5 text-type-entity font-semibold`)
       - No return type shown (per decision)
       - No result count displayed (per decision)

    6. **Result selection path computation:**
       - Entity: `/_api` path to the entity's canonical path. Since entities are navigated through root functions, we need to determine the path. For entity search results, the entity `fullName` is stored but the app navigates via `/_api/{rootFn}/{navProp}/...`. Since the search index stores entities by `fullName` but the app uses function-based routing, for entity results navigate to the first root function whose return type matches the entity (or if no direct root function, a reasonable fallback). **Simpler approach:** Build a reverse lookup during component mount — for each entity with at least one root function that returns it, map entity fullName → `/_api/{rootFnName}`. For entities not directly accessible via a root function, we can skip navigation or use a different approach. Consult the search index `id` format: `entity:SP.Web` → look up which root function returns `SP.Web` (e.g., `web` → `/_api/web`).
       - Function (bound): `id` is `fn:{parentEntity}/{fnId}`. Navigate to `/_api/{path to parent entity}/{fnName}`. Since we have `parentEntity` (the full name), we need to resolve the parent entity's path. **Pragmatic approach:** Use the lookup maps' `entityByFullName` + walk the root functions. For simplicity, compute a `pathMap` at boot that maps each search document ID to its `/_api/...` path. OR — compute paths lazily when results arrive using a helper function that resolves entityFullName → path by finding the root function chain.
       - Function (root): `id` is `fn:root:{fnId}`. Navigate to `/_api/{fnName}`.
       - Nav property: `id` is `nav:{parentEntity}/{navName}`. Navigate to `/_api/{path to parent entity}/{navName}`. Same parent resolution challenge.

       **Implementation strategy for path resolution:**
       Create a helper function `resolveSearchResultPath(result: SearchResult): string` that:
       - Parses the `id` field to determine the type (entity/fn/nav) and extract identifiers
       - For root functions: simply `/_api/{name}`
       - For bound functions: finds the entity's shortest `/_api/...` path by checking which root function returns this entity type (walk: root fn → entity via returnType). Since many entities are 1 hop from root (e.g., `web` returns `SP.Web`, `SP.Web` has `Lists` nav prop returning `SP.List`), a BFS from root functions through nav properties would give shortest paths. **But this is complex.** Simpler: since the search document stores `parentEntity` (the entity fullName the function/nav belongs to), and each entity either IS a root function's return type OR is reachable via nav properties, we can:
         1. Build a `entityToRootPath` map once: for each root function, map `returnType` → `/_api/{fnName}`
         2. For entities reachable via nav properties, recursively build paths via BFS from root
         3. Cache this map in a module-level variable (build once when first needed)
       - For nav properties: `parentEntity` path + `/{navName}`

       **Create a `buildPathMap` utility** in a separate file or within the component. Use the existing `metadata` from `getMetadata()` and `getLookupMaps()`. Build a `Map<string, string>` where key = entity fullName, value = shortest `/_api/...` path to that entity. Then path resolution becomes: look up entity path, append function/nav name if needed.

       Place this utility as a helper function inside CommandPalette.tsx (or a small `search-utils.ts` if it gets large). Use `useMemo` to compute the path map from metadata + lookup maps.

       The BFS should:
       ```
       queue = []
       for each root function:
         if composable and returnType exists:
           pathMap.set(returnType, `/_api/${fn.name}`)
           queue.push({ entityFullName: returnType, path: `/_api/${fn.name}` })
       while queue not empty:
         { entityFullName, path } = queue.shift()
         entity = entityByFullName.get(entityFullName)
         if !entity: continue
         for each navProp of entity.navigationProperties:
           navPath = `${path}/${navProp.name}`
           if !pathMap.has(navProp.typeName):
             pathMap.set(navProp.typeName, navPath)
             queue.push({ entityFullName: navProp.typeName, path: navPath })
       ```

    7. **Keyboard navigation:**
       - cmdk handles ↑/↓ navigation and Enter selection natively
       - Wrap-around: cmdk supports this by default (loop prop or similar). Check cmdk docs — if not, handle manually.
       - Per decision: "No auto-highlight of first result — user must arrow down or click to select." cmdk may auto-select the first item by default. If so, we need to disable this. Use cmdk's `value` prop set to empty string initially, or use a custom approach.
       - "Mouse hover and keyboard share the same highlight state" — cmdk handles this natively.
       - Escape closes the palette — handled by CommandDialog (Radix Dialog dismiss on Escape).

    8. **Footer hint bar:**
       - Below the CommandList, render a footer bar showing keyboard shortcuts:
       - `↑↓ Navigate` · `↵ Open` · `Esc Close`
       - Style: `text-xs text-muted-foreground border-t border-border px-3 py-2 flex items-center gap-3`
       - Use `<kbd>` elements for the key symbols

    9. **Animation:** Per decision — "Subtle open/close animation (fade + slight scale)". The shadcn CommandDialog should handle this via Radix Dialog's enter/exit animations + tw-animate-css classes already in index.css.

    10. **Positioning:** Per decision — "Positioned top-center of viewport (like VS Code, GitHub, Linear)". Override the CommandDialog content positioning to be `top-[20%]` instead of centered, like VS Code's command palette. Apply via className override on the dialog content.
