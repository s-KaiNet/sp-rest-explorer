# Phase 07.2: Add path to API Endpoints index - Context

**Gathered:** 2026-02-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Add path-based search to the API Endpoints search index. Users can search endpoints by their full `_api/...` REST path, not just by leaf name. This phase modifies the search indexing and query behavior — not navigation, routing, or endpoint detail views.

</domain>

<decisions>
## Implementation Decisions

### Search mode trigger
- **Single word without slash or space** → name-only search (current behavior, name index)
- **Query contains `/`** → path search mode (path index)
- **Query contains a space** → path search mode with AND semantics (path index, all terms must match)
- Typing `_api/` alone does NOT trigger path mode (too broad — matches all 62K endpoints). Only non-`_api` slashes trigger it (e.g. `web/`, `Lists/Get`)
- Path mode shows **only the API Endpoints group** — entities group is hidden (entities have no paths)

### Dual-index architecture
- Two separate MiniSearch indexes: one for names (existing), one for paths (new)
- When query triggers path mode, search the path index exclusively
- When query is a single word without slash/space, search the name index only (current behavior preserved)
- Name matches always rank higher than path matches (name index has boost priority when both could apply)

### Path index tokenization
- Tokenize path field on `/` so `_api/web/Lists/GetByTitle` becomes tokens `[_api, web, Lists, GetByTitle]`
- Multi-word space-separated queries use AND combinator: `"lists get"` matches paths containing both `Lists` AND `Get` tokens
- Slash-containing queries like `web/Lists` should match paths containing those segments in sequence

### Path highlighting in results
- When in path mode, highlight the matched segments in the `_api/...` path text (bold/highlight matching portions)
- Same visual treatment as current name highlighting (bold match spans)
- Keep current layout: name primary (top), path secondary (muted text below) — do NOT flip hierarchy in path mode

### Path display format
- Keep full `_api/` prefix in displayed paths — matches what SharePoint developers actually type in REST calls
- Do not strip or abbreviate the path

### Claude's Discretion
- Path index memory optimization (if needed for 62K documents)
- Exact MiniSearch configuration for path index (fuzzy, prefix settings)
- How to handle edge cases: queries that contain both slash AND space
- Highlight implementation approach (reuse existing highlighter or new one for path segments)
- Whether to strip `_api` token from path index to reduce noise

</decisions>

<specifics>
## Specific Ideas

- The slash acts as a natural mode switch — it's an affordance developers already understand from REST paths
- Space-separated words as AND path search is powerful: `"lists get"` → `_api/web/Lists/GetByTitle`
- This is similar to how VS Code's file search works: typing path separators switches from filename search to path search

</specifics>
