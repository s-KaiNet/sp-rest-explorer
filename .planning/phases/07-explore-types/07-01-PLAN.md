Create `app/src/lib/metadata/type-indexes.ts` following the exact same singleton + useSyncExternalStore pattern as `lookup-maps.ts`:

**Type classification heuristic:** A type in `metadata.entities` is a "complex type" (not an entity type) if:
- `navigationProperties.length === 0` AND `functionIds.length === 0`
- AND `fullName` does NOT start with `Collection(` (skip synthetic Collection wrappers)

This heuristic works because entity types always have nav properties or bound functions; complex types only have scalar properties.

**TypeIndexes interface:**
```typescript
interface TypeIndexes {
  /** All complex types, sorted alphabetically by fullName */
  complexTypes: EntityType[]

  /** Map<typeFullName, EntityType[]> — types that have this type as baseTypeName */
  derivedTypes: Map<string, EntityType[]>

  /** Map<typeFullName, UsedByRef[]> — entities whose nav properties reference this type */
  usedByIndex: Map<string, UsedByRef[]>

  /** Namespace groups for sidebar: { namespace: string, types: EntityType[] }[] */
  namespaceGroups: NamespaceGroup[]

  /** Set of all complex type fullNames for O(1) "is this a complex type?" checks */
  complexTypeNames: Set<string>
}

interface UsedByRef {
  entityFullName: string
  entityName: string
  propertyName: string
}

interface NamespaceGroup {
  namespace: string       // e.g. "SP", "SP.Publishing", "SP.Sharing"
  types: EntityType[]     // sorted alphabetically by name within group
}
```

**Build logic (`buildTypeIndexes(metadata, maps)`):**

1. **Classify types:** Iterate `metadata.entities`, apply heuristic, collect complex types. Sort by `fullName`.

2. **Build derivedTypes index:** For every entity in `metadata.entities` (not just complex types — entity types also have base types), if it has `baseTypeName`, add it to `derivedTypes.get(baseTypeName)` array. Sort each array alphabetically.

3. **Build usedByIndex:** For every entity in `metadata.entities` that IS an entity type (has nav props or functions — i.e., NOT a complex type), scan its `navigationProperties`. For each nav property, extract the target type name (unwrap `Collection(...)` if needed). Add a `UsedByRef` to `usedByIndex.get(targetTypeName)`. Sort each array alphabetically. This replaces the O(n*m) scan in UsedByBar with O(1) lookup.

4. **Build namespace groups:** For each complex type, extract namespace from `fullName`:
   - `SP.Web` → namespace `SP`
   - `SP.Publishing.PageLayoutType` → namespace `SP.Publishing`
   - `SP.Sharing.SharedWithMeViewItemRemovalResult` → namespace `SP.Sharing`
   - Rule: everything before the last `.` in `fullName` is the namespace
   Group types by namespace, sort groups by namespace name, sort types within each group by `name`.

5. **Build complexTypeNames Set:** `new Set(complexTypes.map(t => t.fullName))` for O(1) checks.

**Public API (same pattern as lookup-maps.ts):**
- `initTypeIndexes(metadata: Metadata, maps: LookupMaps): void` — build and store singleton, notify listeners
- `getTypeIndexes(): TypeIndexes | null` — non-React access
- `useTypeIndexes(): TypeIndexes | null` — React hook via useSyncExternalStore

**Wire into boot.ts:** In the `hydrate()` function, add `initTypeIndexes(data, getLookupMaps()!)` AFTER `initLookupMaps(data)` and `initSearchIndex(...)`.

**Wire into index.ts barrel:** Export `useTypeIndexes`, `getTypeIndexes`, and the `TypeIndexes`, `UsedByRef`, `NamespaceGroup` types.

**Important:** Do NOT modify the existing `UsedByBar.tsx` in this task — it still works for entity types in Explore API. Plan 02 will handle updating it to use the precomputed index if needed.

4. **Derived types** — query `typeIndexes.derivedTypes.get(type.fullName)`. If non-empty, render a "Derived types" section with each derived type as a clickable `<TypeLink>`. Style as a list of links. Per CONTEXT.md: "Whether to show derived types for complex types (not just entity types): Claude's discretion based on actual inheritance patterns in the data" — show them for all types that have them.

5. **Used by** — query `typeIndexes.usedByIndex.get(type.fullName)`. If non-empty, render "Used by Entities" section. Per CONTEXT.md decisions:
   - Show entities only (no functions) — the index already only has entity nav props
   - Capped with "Show more" toggle — use same pattern as UsedByBar (MAX_VISIBLE = 10, "+N more" button)
   - Each entry is a `<Link to={/entity/${encodeURIComponent(ref.entityFullName)}>` styled as purple chips (same as UsedByBar)
   - Entries show `entityName.propertyName` format
   - Clicking navigates to `/entity/{entityFullName}` which renders EntityDetail on TypesPage.
   
   **Known deviation from CONTEXT.md:** The locked decision says "Clicking a 'Used by' entity navigates to that entity's detail in Explore API." However, there is no existing utility to resolve an entity fullName to an `/_api/` path (entities appear at various depths in the API tree — SP.Web is at `/_api/web` but SP.ListItem might be at `/_api/web/lists/getbytitle('...')/items`). Building a fullName→path resolver would require re-running BFS or maintaining a reverse index, which is scope creep. The `/entity/{fullName}` URL renders the full EntityDetail view (same component used in Explore API), so the user gets complete entity information — just with the types sidebar context instead of the API tree sidebar. This is a pragmatic tradeoff per Claude's discretion.

6. **Properties table** — if `type.properties.length > 0`, render a collapsible section with filter (reuse existing `CollapsibleSection`, `SectionFilter`, and `PropertiesTable` components from `@/components/entity`). Columns: Name, Type (TypeLink), Nullable.

   **Nullable logic per CONTEXT.md:** "a property is nullable unless it explicitly has `"nullable": false` — nullable is the default". The existing `PropertiesTable` already handles this — check that it renders nullable correctly for the case where `property.nullable` is `undefined` (should show as nullable/true).

7. **Empty state** — if no properties, show "No properties defined" in the Properties section.
