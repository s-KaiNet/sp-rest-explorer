# Phase 07.1: Fix Search Experience - Context

**Gathered:** 2026-02-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the search result navigation UX so that clicking any search result reliably navigates to the correct destination. Restructure search indexing to produce two clear groups — Entities (from `entities` JSON) and API Endpoints (from `functions` JSON) — each with correct navigation targets and meaningful path context. Rebuild the API Endpoints index by walking the API tree from `_api/` to a researched depth, storing resolvable paths.

This phase does NOT add new search capabilities (filters, recent searches, etc.). It fixes the existing search so results are correctly grouped, display useful path context, and navigate reliably.

</domain>

<decisions>
## Implementation Decisions

### Result grouping structure
- **Two groups:** "Entities" and "API Endpoints" (not three)
- Fixed display order: Entities first, API Endpoints second
- Groups are collapsible — user can collapse either group
- Collapse state resets on each new search query
- No hard cap on results — show initial batch per group with "Show more..." to expand
- Empty groups are hidden

### Entity group rules
- **Source:** `entities` JSON object only
- **Searchable:** Entity (class) name — both short name (e.g., `Web`) and full OData name (e.g., `SP.Web`)
- **Match rule:** Substring of entity name ONLY. If a search term matches an entity's function name or nav property name but NOT the entity name itself, that entity does NOT appear in this group
- **Display:** Short name as primary text, full OData namespace shown subtly (lighter/smaller) underneath or beside it
- **Badge:** Entity badge
- **Navigation:** Clicking navigates to `/entities/<name>` (entity detail pages already partially exist)
- **Index:** Keep current entity indexing — only filter results to enforce name-only matching

### API Endpoints group rules
- **Source:** `functions` JSON object only (root functions, bound functions, navigation properties)
- **Searchable:** Leaf name only (e.g., `GetById`). Path is displayed but NOT searched
- **Display per result:**
  - Leaf name (the function or nav property name) with search term highlighted
  - Subtle icon/badge distinguishing functions vs nav properties (fn icon vs nav icon)
  - `Root` badge for `isRoot: true` items
  - Full `_api/...` path shown underneath the leaf name — full path displayed, wraps if long
- **Deduplication:** Deduplicate by identity — if the same function on the same entity type is reachable via multiple paths, index it once with the shortest path
- **Duplicates across entities:** If `GetById` exists on SP.List and SP.Web, show ALL of them (different entities = different results)
- **Non-composable roots:** Included in index — they are valid `_api/` leaf endpoints
- **Navigation:** Clicking navigates to `_api/<full path>` — every indexed path must be resolvable by the app's route resolution

### API tree-walk indexing (researcher to investigate)
- **Concept:** Walk the API tree starting from `_api/` root functions, traversing child nodes through composable functions and nav properties
- **Composability rule (CRITICAL):** Only traverse children if the function/nav property has `isComposable: true`. Non-composable items are leaves — the API chain stops there
- **Depth:** NOT locked — researcher must determine optimal depth by analyzing:
  - Performance of client-side tree traversal (time to build index)
  - Recursion/cycle handling (e.g., SP.Web → ParentWeb → SP.Web)
  - Memory footprint (total number of index entries at various depths)
  - Whether a backend pre-computation approach is better (pre-computed index blob on Azure Blob Storage)
- **Backend option:** Open to pre-computing the index on a backend and fetching it as a blob, if researcher finds client-side approach is too heavy
- **Resolvability constraint:** Only index paths that the app can actually resolve and navigate to. If route resolution can't handle a deep path, don't index it
- **Index build timing:** Build on app load in background (after metadata loads), so the index is ready instantly when user opens search

### Keyboard navigation
- Arrow keys move seamlessly through all results across both groups
- Group headers are skipped during keyboard navigation
- Enter selects the highlighted result

### Claude's Discretion
- Exact "Show more..." threshold per group (initial visible count before expand)
- Loading/building state UI while index builds in background
- Exact icon designs for function vs nav property distinction
- How to handle the transition from current 3-group search to new 2-group search (migration approach)

</decisions>

<specifics>
## Specific Ideas

- Two groups map to the app's two main sections: "Explore Types" (entities) and "Explore API" (endpoints) — search results mirror the app's information architecture
- The `isComposable` flag on functions is the key constraint for tree traversal — it determines whether the API chain can continue from a function's return type
- Current BFS approach in `buildEntityPathMap()` is the starting point but needs to be generalized to index all reachable endpoints, not just map entities to their first path
- MiniSearch is already in use and should continue to be the search engine — the change is in what gets indexed and how results are grouped/displayed

</specifics>
