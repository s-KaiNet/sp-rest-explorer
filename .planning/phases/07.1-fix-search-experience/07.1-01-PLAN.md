<objective>
Build the BFS tree-walk algorithm and rewrite the search index to produce two document kinds (entity + endpoint) with pre-computed navigation paths.

Purpose: The current search index has three kinds (entity, function, navProperty) and relies on runtime path resolution via entityPathMap — which fails for most entities and produces broken navigation. This plan replaces the entire indexing backend with a tree-walk that pre-computes every reachable `_api/` path and stores it directly in the search document, eliminating runtime resolution entirely.

Output: New `api-tree-walk.ts` module, rewritten `search-index.ts`, updated `types.ts` with new SearchDocument shape, updated boot sequence.
</objective>

**1. Update `types.ts` — new SearchDocument shape:**

Replace the existing `SearchDocument` interface with a new one that supports both entity and endpoint documents:

```typescript
export interface SearchDocument {
  id: string
  name: string
  fullName: string
  kind: 'entity' | 'endpoint'
  // Endpoint-specific fields (undefined for entities)
  path?: string           // Full _api/... path for display and navigation
  endpointKind?: 'function' | 'navProperty'
  parentEntity?: string   // Entity fullName the leaf belongs to
  isRoot?: boolean        // Whether this is a root function
}
```

Also export a new `EndpointEntry` interface (used internally by the tree-walk):

```typescript
export interface EndpointEntry {
  id: string              // "endpoint:{parentEntity}|{leafName}|{kind}" for dedup
  name: string            // Leaf name (searchable)
  path: string            // Full _api/... path (displayed, not searched)
  kind: 'function' | 'navProperty'
  parentEntity: string    // Entity fullName the leaf belongs to
  isRoot: boolean         // Whether this is a root function
}
```

**2. Create `api-tree-walk.ts` — BFS tree-walk algorithm:**

Implement `buildApiEndpointIndex(metadata: Metadata, lookupMaps: LookupMaps): EndpointEntry[]` following the research-verified algorithm:

- **Seed phase:** Iterate ALL root functions (`fn.isRoot === true`). For each:
  - Create an EndpointEntry with `id: "ROOT|{fn.name}|function"`, `path: "_api/{fn.name}"`, `isRoot: true`, `parentEntity: ""`
  - Add to a `Map<string, EndpointEntry>` keyed by identity string (for dedup)
  - If `fn.isComposable && fn.returnType && entityByFullName.has(fn.returnType)`: add the return type entity to a BFS queue (if not already in `visitedEntities` Set), mark as visited

- **BFS phase:** Process queue entries `{ entityFullName, path }`:
  - Get the entity from `entityByFullName`
  - For each nav property: create entry with `id: "{entityFullName}|{nav.name}|navProperty"`, check dedup map, if entity `nav.typeName` exists and not visited → add to queue + visited
  - For each bound function (via `entity.functionIds` + `functionById`): create entry with `id: "{entityFullName}|{fn.name}|function"`, check dedup map, if `fn.isComposable && fn.returnType` exists and not visited → add to queue + visited

- **Dedup logic:** For identity collisions, keep the entry with the shorter `path` (per CONTEXT.md: "deduplicate by identity — same function on same entity type is reachable via multiple paths, index it once with the shortest path")

- **Cycle detection:** Entity-level visited Set prevents infinite loops (e.g., SP.Web → ParentWeb → SP.Web). No depth limit needed — tree converges naturally at depth ~8.

- **Return** `Array.from(indexed.values())`

Export both `buildApiEndpointIndex` and the `EndpointEntry` type.

**1. Rewrite `search-index.ts`:**

Replace the entire `buildSearchIndex` function. The new version accepts both Metadata AND LookupMaps:

```typescript
export function buildSearchIndex(
  metadata: Metadata,
  lookupMaps: LookupMaps,
): MiniSearch<SearchDocument>
```

MiniSearch configuration:
- `fields: ['name', 'fullName']` — Both fields indexed for search. Entity docs use both (short name + full OData name). Endpoint docs set `fullName` to `''` (empty string) so fullName doesn't contribute to endpoint search — per CONTEXT.md "Searchable by leaf name only, path displayed but NOT searched".
- `storeFields: ['name', 'fullName', 'kind', 'path', 'parentEntity', 'isRoot', 'endpointKind']`
- `searchOptions: { boost: { name: 2 }, fuzzy: 0.2, prefix: true }`
- `tokenize: (text) => text.split(/[\s._]+/)` — unchanged, splits on spaces, dots, underscores

Document building:
1. **Entity documents:** Loop `Object.values(metadata.entities)`. For each entity:
   ```
   { id: `entity:${entity.fullName}`, name: entity.name, fullName: entity.fullName, kind: 'entity' }
   ```
   Per CONTEXT.md: Entity name-only matching. The user searches "Web" and gets SP.Web or "SP.Web" and gets it via fullName. Entities whose FUNCTION or NAV PROPERTY names match should NOT appear — only entity name matching.

   **IMPORTANT:** MiniSearch doesn't natively support "this document should only match on field X." Since we index `name` and `fullName` for ALL docs, and endpoint docs have `fullName: ''`, this naturally gives entities both-field search and endpoints name-only search. Correct behavior.

2. **Endpoint documents:** Call `buildApiEndpointIndex(metadata, lookupMaps)` to get EndpointEntry[]. For each entry:
   ```
   {
     id: `endpoint:${entry.id}`,
     name: entry.name,
     fullName: '',  // Empty — path NOT searchable per CONTEXT.md
     kind: 'endpoint',
     path: entry.path,
     endpointKind: entry.kind,
     parentEntity: entry.parentEntity,
     isRoot: entry.isRoot,
   }
   ```

3. Call `index.addAll(docs)`, log count: `console.log('[SP Explorer] Search index:', docs.length, 'items indexed (', entityCount, 'entities,', endpointCount, 'endpoints)')`

Update `initSearchIndex` signature to accept LookupMaps:
```typescript
export function initSearchIndex(metadata: Metadata, lookupMaps: LookupMaps): void {
  searchIndex = buildSearchIndex(metadata, lookupMaps)
}
```
