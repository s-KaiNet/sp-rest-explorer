<task type="auto">
  <name>Task 1: Fix nullable column to use strict equality check</name>
  <files>app/src/components/entity/PropertiesTable.tsx</files>
  <action>
In PropertiesTable.tsx, the current nullable display logic uses a truthy check:
```tsx
{prop.nullable ? (
  <span className="text-muted-foreground">yes</span>
) : (
  <span className="font-semibold text-type-fn">no</span>
)}
```

The Property interface defines `nullable?: boolean` (optional). When metadata omits the `nullable` field entirely, `prop.nullable` is `undefined` — which is falsy — so it incorrectly renders "no".

Per user decision: `nullable: false` → "no"; everything else (true, undefined, missing) → "yes".

Change the condition to use strict equality:
```tsx
{prop.nullable === false ? (
  <span className="font-semibold text-type-fn">no</span>
) : (
  <span className="text-muted-foreground">yes</span>
)}
```

Note the order flip: the `false` case (showing "no") comes first, then the default (showing "yes"). This ensures:
- `nullable: false` → "no"
- `nullable: true` → "yes"
- `nullable: undefined` (missing) → "yes"

Display format: plain lowercase text ("yes" / "no") — no icons, badges, or colors beyond existing styling. Keep the existing className styling for each case.
  </action>
  <verify>
Run `npm run build` from the `app` directory — build must succeed with no type errors.

Manual verification: In the running app, navigate to any entity with properties (e.g., `/_api/web` → click an entity). Check that:
1. Properties with `nullable: false` in the raw metadata show "no"
2. Properties with `nullable: true` or no nullable field show "yes"
  </verify>
  <done>
The nullable column accurately reflects the metadata: "no" only when nullable is explicitly false, "yes" in all other cases. Build passes.
  </done>

**Current state:** In ExplorePage.tsx, the BreadcrumbBar renders at the top of the component, spanning full width above both the sidebar and content area (line 102). It appears on ALL views including the root listing page.

**Required changes:**

### 1. ExplorePage.tsx — Move breadcrumb into content area, conditional on non-root

Remove the BreadcrumbBar from its current position (above the sidebar+content flex container). Instead, render it INSIDE the content area div (the right panel, `<div className="flex-1 overflow-y-auto bg-muted/30">`), but only when `!isRoot` (detail pages only).

The breadcrumb should render ABOVE the ContentTransition, inside the scrollable content area. But since it needs to be sticky, it must be inside the scrollable container but with `position: sticky; top: 0`.

Updated structure for the content area:
```tsx
{/* Right: Content area with independent scroll */}
<div className="flex-1 overflow-y-auto bg-muted/30">
  {/* Sticky breadcrumb — only on detail pages */}
  {!isRoot && segments.length > 0 && (
    <BreadcrumbBar segments={segments} onNavigate={handleBreadcrumbNavigate} />
  )}
  <ContentTransition pathKey={pathKey}>
    {/* ... existing content rendering ... */}
  </ContentTransition>
</div>
```

The breadcrumb should NOT appear on the root listing page (isRoot === true). It SHOULD appear for:
- Entity detail pages (currentEntity is truthy)
- Function detail pages (currentFunction is truthy)
- The "not found" state (still shows the path they tried)

### 2. BreadcrumbBar.tsx — Add sticky positioning and bottom border

Update the BreadcrumbBar's nav element className to be sticky:

Current:
```tsx
className="group z-10 flex min-h-[40px] shrink-0 flex-wrap items-center border-b border-border bg-sidebar px-4 py-2"
```

Change to:
```tsx
className="group sticky top-0 z-10 flex min-h-[40px] flex-wrap items-center border-b border-border bg-sidebar px-4 py-2"
```

Key changes:
- Add `sticky top-0` — pins breadcrumb at top of scrollable content area when scrolling
- Remove `shrink-0` — not needed when sticky
- Keep `flex-wrap` — per user decision, long breadcrumb paths wrap to next line rather than truncating
- Keep existing `bg-sidebar` — ensures the sticky bar has an opaque background so content doesn't show through
- Keep existing `border-b border-border` — this serves as the subtle bottom border the user wants

No other styling changes needed — the breadcrumb component already has the right visual treatment. The fix is purely positional (moving it in the DOM) and behavioral (sticky + conditional).
